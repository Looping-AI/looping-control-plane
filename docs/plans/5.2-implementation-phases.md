# 5.2 ‚Äî Execution Engines + Scope System: Implementation Phases

## Pre-Implementation Note: Envelope Is Pure Data, Access Is Scoped

The `ExecutionEnvelope` is a **serializable, data-only artifact** ‚Äî arrays, records, and primitives. It never contains `Map.Map`, `var` fields, closures, or any internal state representation. An **`EngineEnvelopeBuilder`** adapter converts internal state (Maps, session stores, channel history) into the envelope's serializable structures: `[ChatMessage]`, `Text`, `[(Text, Text)]`, etc.

The envelope carries **scope grants** ‚Äî declarations of what the engine is allowed to access (e.g., `workspaces:read`, `agents:write`). It does NOT carry serialized state. State access happens through a **State Gateway** ‚Äî a controlled, token-validated interface that the orchestrator constructs and passes alongside the envelope.

**The `#internal` engine has no direct access to canister state.** It receives:

1. An **envelope** (pure data: messages, instructions, constraints, secrets, scope grants)
2. A **State Gateway** (record of typed accessor functions that validate scope + token before executing reads/writes against canonical state)
3. An **execution token** (ephemeral nonce bound to scope grants, revoked after execution completes)

The engine's tool handlers call the gateway for all state operations. The gateway validates every request against the token's scope grants before executing. The engine never sees `Map.Map`, `var` fields, or model-level state references ‚Äî it interacts with canonical state exclusively through the authorized, auditable gateway.

This pattern is universal: for `#internal`, the gateway is function closures over canonical state; for external engines (future), the gateway is an HTTP callback endpoint with a cryptographically signed token. The engine boundary is an authorization boundary.

## Phase 1 ‚Äî Execution Types

**Goal:** Establish the shared type vocabulary. Zero behavior, pure type definitions.

**Create:**

- `types/execution.mo`
  - `ScopeResource` ‚Äî variant: `{ #workspaces; #agents; #secrets; #eventStore; #sessionPolicy }`
  - `ScopeAccess` ‚Äî variant: `{ #read; #write }` (write implies read)
  - `ScopeGrant` ‚Äî `{ resource : ScopeResource; access : ScopeAccess }`
  - `ExecutionEnvelope` ‚Äî `{ envelopeVersion, requestId, agentId, agentName, workspaceId, workflowId, messages, instructions, constraints, secrets, scopeGrants }`
  - `ExecutionPackage` ‚Äî `{ packageVersion, requestId, status, response, trace, stats, availableWorkflows }`
  - `PackageStatus` ‚Äî `{ #completed; #failed : Text; #roundLimitReached }`
  - `ExecutionStats` ‚Äî `{ durationNs, llmCalls, toolCalls, inputTokens, outputTokens, model, rounds }`
  - `ExecutionConstraints` ‚Äî `{ maxRounds : Nat; maxTokenBudget : ?Nat }`
  - `TraceEntry` ‚Äî reuse/alias existing `SessionModel.TurnTraceEntry` structure, or define engine-local trace type
  - `WorkflowCatalog` helpers (if needed)

**Design decisions:**

- `TraceEntry` in the Package should be structurally compatible with `SessionModel.TraceDetail` so the Result Applicator can write traces without transformation. Consider making them the same type or defining a mapping.
- `secrets` field: `{ apiKeys : [(Text, Text)] }` ‚Äî flat key-value pairs, no closures, no encrypted blobs.
- `messages` field: reuse `OpenRouterWrapper.ResponseInputMessage` or define engine-local `ChatMessage` type? If engine-local, it must be convertible from the wrapper's type during envelope construction.
- `scopeGrants` field: declarative only ‚Äî `[ScopeGrant]` tells the engine what scopes it has been granted. The engine doesn't use these directly to access state ‚Äî it uses the State Gateway (Phase 3) which was constructed from these grants. The grants in the envelope serve as documentation/audit trail and for the engine to know which tools it can offer.
- **No `stateSnapshots`** ‚Äî state is accessed on-demand through the gateway, not serialized into the envelope.
- **No `effects`** ‚Äî state mutations happen through the gateway during execution, not deferred to a post-execution applicator.

**Checkpoint:** `icp build control-plane-core` ‚Äî types compile, no consumers yet.

**Estimated scope:** ~80‚Äì120 lines.

## Phase 3 ‚Äî Scope System + State Gateway

**Goal:** Establish the scope-based access control system that governs how engines interact with canonical state. No direct state access ‚Äî all operations go through a validated State Gateway.

**Motivation:**

The existing `ToolResources` already has the right concept ‚Äî each resource is optional (`?{ state; write : Bool }`), creating a natural allowlist where omitting a resource prevents tools from accessing it. The problem is that `ToolResources` passes mutable state references (`Map.Map`, `SessionStores`) directly to the engine. The scope system formalizes this into:

1. **Scope grants** ‚Äî declarative `(resource, access)` pairs in the envelope (pure data)
2. **Execution tokens** ‚Äî ephemeral per-turn credentials for authentication + revocation + audit
3. **State Gateway** ‚Äî the controlled interface that validates scope + token before executing any state operation

This is the same allowlist pattern, but instead of passing state refs with a `write` flag, we pass authorized functions with the authorization already baked in.

### 3a. `services/execution-token-service.mo` ‚Äî token lifecycle

**Create:**

- `TokenStore` ‚Äî `Map<Text, TokenRecord>` keyed by nonce
- `TokenRecord` ‚Äî `{ nonce : Text; turnId : Text; grants : [ScopeGrant]; createdAtNs : Int; expiresAtNs : Int; revoked : Bool; parentNonce : ?Text }`
- `issue(turnId : Text, grants : [ScopeGrant], ttlNs : Int) ‚Üí Text` ‚Äî generates a random nonce, stores the record, returns the nonce
- `validate(nonce : Text, resource : ScopeResource, access : ScopeAccess) ‚Üí Bool` ‚Äî checks: nonce exists, not expired, not revoked, and grants cover `(resource, access)`. A `#write` grant implies `#read`.
- `revoke(nonce : Text)` ‚Äî marks the token as revoked. Cascades: also revokes all child tokens derived from this nonce.
- `derive(parentNonce : Text, narrowedGrants : [ScopeGrant]) ‚Üí Result<Text, Text>` ‚Äî for sub-agent delegation. The derived token's grants must be a **subset** of the parent's. Returns `#err` if attempting to escalate. Child token's `parentNonce` points to the parent.
- `cleanup()` ‚Äî removes expired/revoked tokens. Called on each `issue()` or periodically.

**Token design decisions:**

- **`#internal` tokens are simple nonces.** Engine and gateway share the same Wasm instance ‚Äî the nonce is a random unique ID that indexes into the `TokenStore`. No cryptographic signing needed.
- **External engine tokens (future) would be HMAC-signed:** `HMAC(canister_secret, nonce || grants || expiry)`. The external engine presents this signed token to the HTTP gateway endpoint. The gateway can validate without a store lookup (stateless verification).
- **Token TTL** is a safety net (e.g., 5 minutes). The orchestrator also explicitly revokes after execution completes. Belt and suspenders.
- **Sub-agent delegation:** when an agent delegates to a sub-agent, the orchestrator derives a child token with equal or narrower scopes. This is how scope attenuation works ‚Äî a `workspaces:write` parent can delegate `workspaces:read` to a child. Revoking the parent cascades to all children.

### 3b. `engines/internal/state-gateway.mo` ‚Äî controlled state interface

The State Gateway is a **record of typed, per-resource accessor functions**. Each resource scope is optional (`null` = scope not granted). Within each resource scope, write operations are separately optional (`null` = read-only grant). This mirrors the existing `ToolResources` gating pattern ‚Äî `null`-check on the resource = scope check ‚Äî but replaces raw state refs with validated gateway functions.

```motoko
type StateGateway = {
  // ‚îÄ‚îÄ Per-resource scopes (null = not granted) ‚îÄ‚îÄ
  workspaces : ?WorkspaceScope;
  agents : ?AgentScope;
  secrets : ?SecretScope;
  eventStore : ?EventStoreScope;
  sessionPolicy : ?SessionPolicyScope;

  // ‚îÄ‚îÄ Auxiliary context (non-state, carried for tool handlers) ‚îÄ‚îÄ
  userAuthContext : ?UserAuthContext;
  triggerMessageText : ?Text;
  resolveSlackBotToken : ?(Text -> ?Text);
  openRouterApiKey : ?Text;
  workspaceId : ?Nat;
};

type WorkspaceScope = {
  // Read operations ‚Äî always present when scope is granted
  list : () -> async Text; // ‚Üí JSON array of workspaces
  get : (Nat) -> async Text; // ‚Üí JSON workspace or "not found"
  // Write operations ‚Äî null if scope is read-only
  create : ?(CreateWorkspaceArgs -> async Text);
  delete : ?(DeleteWorkspaceArgs -> async Text);
  setAdminChannel : ?(SetAdminChannelArgs -> async Text);
};

// AgentScope, SecretScope, etc. follow the same pattern

```

**How it works:**

Each gateway function is a closure constructed by the orchestrator. The closure captures:

1. The execution token nonce
2. A reference to the token service (for per-call validation)
3. The canonical state reference (the actual `Map.Map` or model state)

On every call, the closure:

1. Validates the token (not expired, not revoked, scope still valid)
2. Executes the model-level operation (read or write)
3. Serializes the result to JSON
4. Returns it to the engine

If validation fails, the closure returns an error. The engine never learns why validation failed (no information leakage about other scopes).

**Why per-resource typed functions instead of a generic `gateway.query(resource, operation, args)` API?**

- **Type safety**: handlers know exactly what functions are available at compile time
- **Same gating pattern**: `null`-check = scope check (mirrors `ToolResources`)
- **No stringly-typed dispatch**: operations are function calls, not string routing
- **Resource-specific signatures**: each operation has its own argument/return types
- **Natural for tool handlers**: `list_workspaces` calls `gateway.workspaces?.list()` ‚Äî clear and direct

### 3c. `services/gateway-builder.mo` ‚Äî gateway construction

- `buildGateway(tokenService, nonce, grants, stateRefs, auxContext) ‚Üí StateGateway`
- Iterates `grants`:
  - For each resource with `#read` or `#write` ‚Üí creates the resource scope with read functions
  - For each resource with `#write` ‚Üí additionally wires write functions
  - For non-granted resources ‚Üí `null`
- `stateRefs` is a record of canonical state references (only the orchestrator touches this ‚Äî the gateway builder runs at the orchestrator level, not inside the engine)
- `auxContext` carries `userAuthContext`, `triggerMessageText`, `resolveSlackBotToken`, etc.

**Checkpoint:** `icp build control-plane-core` + unit tests:

- Issue token with `[workspaces:read, agents:write]` ‚Üí validate `workspaces:read` ‚Üí `true`
- Validate `workspaces:write` ‚Üí `false` (only read granted)
- Validate `agents:write` ‚Üí `true`
- Validate `secrets:read` ‚Üí `false` (not granted)
- Revoke token ‚Üí all validations ‚Üí `false`
- Derive child with `[agents:read]` from parent `[workspaces:read, agents:write]` ‚Üí succeeds
- Derive child with `[secrets:read]` ‚Üí error (not in parent's grants)
- Build gateway with `workspaces:read` ‚Üí `gateway.workspaces` = `?{ list, get, create = null, delete = null, ... }`
- Build gateway without workspaces scope ‚Üí `gateway.workspaces` = `null`

**Estimated scope:** ~300‚Äì500 lines (token service + gateway types + builder).

## Phase 4 ‚Äî Internal Engine: Tool System

**Goal:** The engine's own tool registry, executor, and handler implementations ‚Äî no imports from `tools/`.

**Create:**

- `engines/internal/tool-types.mo` ‚Äî engine-local tool types
  - `ToolResult`, `ToolHandler` types
  - Handlers receive the `StateGateway` ‚Äî not `ToolResources`, not state refs
- `engines/internal/tool-registry.mo` ‚Äî workflow-based tool selection
  - `getTools(workflowId, gateway) ‚Üí [Tool]` ‚Äî returns tool definitions for a workflow, gated by which gateway resource scopes are present (`null`-check on `gateway.workspaces`, `gateway.agents`, etc. ‚Äî same pattern as today's `ToolResources`)
  - `getHandler(workflowId, toolName) ‚Üí ?ToolHandler` ‚Äî returns the handler for a specific tool
  - Workflow `"admin-v1"`: same tools as current `FunctionToolRegistry.getAll(resources)`
  - Workflow `"onboarding-v1"`: stub, returns empty tool set
- `engines/internal/tool-executor.mo` ‚Äî tool dispatch
  - `execute(workflowId, gateway, toolCalls) ‚Üí async [ToolExecResult]` ‚Äî executes tool calls through the gateway
  - `formatResultsForLlm(results) ‚Üí Text`
- `engines/internal/handlers/` ‚Äî tool handler modules
  - Port each handler from `tools/handlers/` into the engine's handlers directory
  - **Read handlers** call gateway read functions (e.g., `list_workspaces` calls `gateway.workspaces?.list()`)
  - **Write handlers** call gateway write functions (e.g., `create_workspace` calls `gateway.workspaces?.create?(args)`)
  - State mutations happen immediately through the gateway ‚Äî no deferred effects, no shadow state
  - Multi-round consistency is natural: round 2 reads see round 1's writes because the gateway operates on canonical state
  - Handlers import from `models/execution-types.mo` for shared types ‚Äî not from `tools/`, `services/`, or `wrappers/`

**Key challenge ‚Äî tool handler migration scope:**

The current `tools/handlers/` directory contains all admin tool handlers. Each handler is typically 50‚Äì200 lines. The engine needs its own copies because:

1. The engine can't import from `tools/handlers/`
2. Handlers now call gateway functions instead of accessing state directly (different interface)
3. Different workflows may have different tool implementations

The handlers are SIMPLER in the new system ‚Äî they don't capture state, they call gateway methods. A typical handler: (1) parse JSON arguments, (2) call the appropriate gateway function, (3) return the result.

Pragmatic approach for this PR:

- **High-value tools first**: port the most-used admin tools (workspace CRUD, agent CRUD, secret management)
- **Stub remaining tools**: return "not yet available in this engine version" for rarely-used tools
- **Follow-up PR**: complete the tool migration

**State access pattern ‚Äî ALL engine types, including `#internal`:**

- The engine has **zero direct access** to canonical canister state. No state references, no `Map.Map` fields, no closures over state.
- **Read operations** go through gateway functions that validate scope + token before executing.
- **Write operations** go through gateway functions that validate scope + token before executing. Writes happen immediately against canonical state ‚Äî the tool gets the real result (actual IDs, actual state) for the LLM.
- The **State Gateway** (Phase 3) is the sole interface between the engine and canonical state.
- This is identical in concept for `#internal`, `#canister`, and external engines ‚Äî only the transport mechanism differs (closures vs HTTP).

**Checkpoint:** `icp build control-plane-core` ‚Äî tool system compiles, tools can be invoked in isolation.

**Estimated scope:** ~600‚Äì1,000 lines (registry + executor + handler ports). Handlers are simpler than current versions since they delegate to gateway.

## Phase 5 ‚Äî Internal Engine: Loop Assembly

**Goal:** Wire Phases 2 + 4 into the complete engine module with the multi-round LLM loop.

**Create:**

- `engines/internal/internal-execution-engine.mo`
  - `execute(envelope : ExecutionEnvelope, gateway : StateGateway) ‚Üí async ExecutionPackage`
  - **Loop logic** (adapted from `org-admin-agent.mo` lines 170‚Äì310):
    1. Select tools + model based on `envelope.workflowId` + gateway scopes
    2. Initialize local trace buffer (`List<TraceEntry>`)
    3. Loop up to `envelope.constraints.maxRounds`:
       - Call `LlmWrapper.reason(apiKey, messages, model, instructions, tools)`
       - If `#textResponse` ‚Üí append `#llmCall` trace, seal Package with `#completed`
       - If `#toolCalls` ‚Üí append `#llmCall` trace, execute tools through gateway, append `#toolCall` traces, add results to messages, continue
       - If `#err` ‚Üí append `#faultRecovered` trace, seal Package with `#failed`
    4. If loop exhausts rounds ‚Üí append `#roundLimitHit`, seal Package with `#roundLimitReached`
    5. Always include `availableWorkflows` in the Package
  - `getAvailableWorkflows() ‚Üí [Text]` ‚Äî returns `["admin-v1", "onboarding-v1"]`

**Design decisions:**

- The engine's `execute` signature is `(ExecutionEnvelope, StateGateway) ‚Üí async ExecutionPackage`. For `#internal`, the gateway is a function parameter. For external engines (future), the gateway is implicit (HTTP callback URL in the envelope metadata).
- `TraceEntry` accumulation is local (a `List` in the engine's scope). No `SessionModel.appendTrace` calls inside the engine. The full trace is returned in the Package.
- Multi-round consistency is automatic: the gateway operates on canonical state, so round 2's reads see round 1's writes. No shadow state needed.
- The Package has no `effects` field ‚Äî state mutations already happened through the gateway during execution.

**Checkpoint:** `icp build control-plane-core` + unit tests:

- Mock LLM ‚Üí `#textResponse`: Package has `#completed`, response, trace with 1 `#llmCall`
- Mock LLM ‚Üí `#toolCalls` then `#textResponse`: trace has `#llmCall` + `#toolCall` + `#llmCall` in order
- Mock LLM ‚Üí loop `maxRounds`: Package has `#roundLimitReached`
- Unknown `workflowId`: Package has `#failed` + `availableWorkflows`

**Estimated scope:** ~200‚Äì300 lines (loop logic + wiring).

## Phase 6 ‚Äî Result Applicator

**Goal:** Single module that applies all post-execution side effects from a Package. (Renamed from "Effect Applicator" ‚Äî tool-level state mutations now happen through the gateway during execution, not as deferred effects.)

**Create:**

- `services/result-applicator.mo`
  - `apply(ctx, turnId, package, slackCtx) ‚Üí async Result<(), Text>`
  - **Trace persistence:** iterate `package.trace`, call `SessionModel.appendTrace()` for each entry
  - **Slack reply:** call `SlackWrapper.postMessage(botToken, channel, response, threadTs, replyMetadata)`
  - **Turn finalization:** call `SessionModel.completeTurn()` with status derived from `package.status`
  - **Cost aggregation:** compute from `package.stats` or from trace `#llmCall` entries
  - **Token revocation:** call `ExecutionTokenService.revoke(nonce)` to invalidate the execution token
  - **Follow-up events:** emit delegation triggers if applicable (future)

**Input types:**

```
ctx : {
  sessionStores : SessionModel.SessionStores;
  // other stores as needed
};
slackCtx : {
  botToken : Text;
  channel : Text;
  threadTs : ?Text;
  agentName : Text;
  agentId : Nat;
  triggerTs : Text;
  triggerChannel : Text;
};
```

**Design decisions:**

- The applicator receives the Package as a sealed, read-only value. It never modifies it.
- **No tool effects to apply.** State mutations happened through the gateway during execution. The applicator handles only post-execution concerns: traces, Slack reply, turn finalization, cost aggregation, token cleanup.
- **Application order:** traces ‚Üí Slack reply ‚Üí turn finalization ‚Üí token revocation.
- Slack post failure: applicator returns `#err`, caller marks turn `#failed`.
- Trace writing is non-fallible (in-memory map operations).
- The applicator does NOT handle channel history persistence ‚Äî that stays in the message handler (persisting the incoming message is a pre-execution concern, not a post-execution effect).

**Checkpoint:** `icp build control-plane-core` + unit tests:

- Given Package with N traces ‚Üí all N written to session model
- Given Package with response ‚Üí Slack post called with correct params
- Given Package `#failed` ‚Üí turn marked `#failed`
- Given Package `#completed` but Slack fails ‚Üí returns `#err`

**Estimated scope:** ~100‚Äì180 lines (simpler than original ‚Äî no tool effect application).

## Phase 7 ‚Äî Orchestrator + Handler Rewiring

**Goal:** Replace the current dispatch chain with envelope ‚Üí engine ‚Üí applicator.

**Modify:**

### 7a. `orchestrators/agent-orchestrator.mo`

Current flow:

```
orchestrateAgentTalk(agent, ...) ‚Üí resolve API key ‚Üí OrgAdminAgent.process(agent, ...) ‚Üí return response
```

New flow:

```
orchestrateAgentTalk(agent, ...) ‚Üí
  1. Resolve API key (stays here ‚Äî guard rail, not execution logic)
  2. Determine scope grants for the agent (from agent config / workflow)
  3. Issue execution token: ExecutionTokenService.issue(turnId, grants, ttl)
  4. Build StateGateway: GatewayBuilder.build(tokenService, nonce, grants, stateRefs, auxContext)
  5. Build ExecutionEnvelope:
     - InstructionComposer.compose() ‚Üí instructions
     - ContextAssembler.assemble() ‚Üí messages
     - constraints from agent config or defaults
     - secrets: { apiKeys: [("openrouter", apiKey)] }
     - workflowId from cached catalog (default: "admin-v1")
     - scopeGrants (for audit trail / engine tool selection)
  6. Call InternalExecutionEngine.execute(envelope, gateway)
  7. Update cached workflow catalog from package.availableWorkflows
  8. Return package to caller for result application
  9. (Token revocation happens in Result Applicator or here as cleanup)
```

The orchestrator no longer calls `OrgAdminAgent.process()`. Context assembly + instruction composition move here (or into a dedicated envelope builder).

### 7b. `events/handlers/message-handler.mo`

Current flow:

```
handle(msg, ctx) ‚Üí ... ‚Üí dispatchToAgentRouter() ‚Üí postAgentReply() ‚Üí completeTurn()
```

New flow:

```
handle(msg, ctx) ‚Üí ... ‚Üí dispatchToAgentRouter() ‚Üí
  receive (package, slackCtx) from orchestrator ‚Üí
  ResultApplicator.apply(ctx, turnId, package, slackCtx) ‚Üí
  completeTurn based on applicator result
```

Key changes:

- `postAgentReply()` removed ‚Äî Slack posting moves to Result Applicator
- `dispatchToAgentRouter()` returns a Package instead of `(steps, replyText, isError)`
- Turn completion logic simplified ‚Äî derives status from Package + applicator result

### 7c. `agents/admin/org-admin-agent.mo`

The `process()` function is **deprecated/removed**. Its responsibilities are split:

- Envelope construction (instructions, context, scope grants) ‚Üí orchestrator (Phase 7a)
- Gateway construction (token + scope validation) ‚Üí orchestrator (Phase 7a)
- LLM loop + tool execution ‚Üí engine (Phases 2‚Äì5)
- Trace writing + Slack posting ‚Üí Result Applicator (Phase 6)

What remains in `org-admin-agent.mo`:

- `AdminCtx` type definition (still needed by orchestrator)
- `buildInstructions()` helper (or move to orchestrator)
- Eventually this file may be reduced to just the type definition, or merged into the orchestrator

### 7d. Rename `#api` ‚Üí `#internal`

- `models/agent-model.mo`: `ExecutionEngine` variant `#api` ‚Üí `#internal`
- All pattern matches and references throughout the codebase
- Any test fixtures or cassettes that reference the engine type

### 7e. `events/agent-router.mo`

Minimal changes ‚Äî the router still gates by channel allowlist and category. The orchestrator it delegates to now returns a Package instead of a response string. The router may need its return type updated, or the orchestrator can unwrap the Package into the existing return shape to minimize blast radius.

**Checkpoint:** `icp build control-plane-core` + `mops test` + `bun run tsc --noEmit`

**Estimated scope:** ~300‚Äì500 lines of modifications across 4‚Äì5 files.
