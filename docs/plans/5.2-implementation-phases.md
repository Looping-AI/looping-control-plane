# 5.2 — Workflow Engines + Execution API: Implementation Phases

## Pre-Implementation Note: Engine Is a Separate Canister

The `#internal` workflow engine runs in a **separate canister** — a Motoko actor class spawned and lifecycle-managed by the Core canister. This is NOT a virtual/inline engine. It is a distinct Wasm instance with its own principal, cycle balance, and upgrade path.

**What the engine canister is:**

- A **Motoko actor class** (`InternalEngine`) that Core spawns with `await (with cycles) InternalEngine.InternalEngine(coreId)`
- A **stateless executor** — no meaningful persistent state between executions. It receives an envelope, runs the LLM loop, and returns a result.
- Has its own **HTTP outcall capability** — transform function defined directly on the engine actor (resolves the `shared query` constraint)

**What Core manages:**

- **Spawning**: Core creates the engine canister on first need (lazy init), passing its own principal as the `coreId` constructor argument
- **Top-up**: A timer-based runner (following the existing runner pattern) checks the engine's cycle balance periodically and tops it up if below threshold
- **Upgrade propagation**: On Core's `postupgrade`, a one-shot timer (1 min delay) upgrades the engine canister to the latest Wasm via `(system InternalEngine.InternalEngine)(#upgrade engine)(coreId)`

**Communication pattern — engine calls Core:**

The engine calls Core through a **single `shared` update endpoint** (`executionApi`) that receives a JSON body. Core routes the request through an `ExecutionApiService` that parses the body, validates caller + token + scope, dispatches to the appropriate handler, and returns a JSON response. The same service module handles both the inter-canister path and the future HTTP webhook path (`/api/execution/`) — no code duplication.

Two categories of operations flow through this endpoint:

1. **Effects** — scoped state operations (workspace CRUD, agent CRUD, secret management, etc.). The engine presents its execution token with each call. Core validates the caller principal + token scope grants before executing.
2. **Events** — `#executionMilestone` and `#executionComplete`. Core's event handler posts to Slack (with 15s debounce for milestones) and manages turn lifecycle (`completeTurn` on `#executionComplete`).

**What does NOT cross the canister boundary:**

- **Traces** — what happens in the engine, stays in the engine. No `traceWrite` callback. Detailed execution internals (LLM call timing, raw tool outputs, error recovery attempts) are the engine's own concern.
- **State snapshots** — state is accessed on-demand through the execution API, not serialized into the envelope.

The `ExecutionEnvelope` remains **pure serializable data** — messages, instructions, constraints, secrets, scope grants. It also carries the `envelopeNonce` so the engine knows how to authenticate its calls back to Core.

This pattern is universal:

- For `#internal`: the engine calls Core's `shared` `executionApi` endpoint (inter-canister call)
- For `#canister` (future): same inter-canister pattern, different canister (not spawned by Core)
- For external engines (future): HTTP calls to Core's `/api/execution/` — same `ExecutionApiService` handles the request body

The engine boundary is an **authorization + process boundary**. The execution token authenticates every call. For the inter-canister path, Core also validates `msg.caller == enginePrincipal` as a transport-level guard in `main.mo` (before the request reaches the service). For the HTTP path, transport-level auth uses a different mechanism (bearer token / API key). The `ExecutionApiService` itself only does token-level validation — it doesn't know or care about transport.

## Phase 1 — Shared Execution Types

**Goal:** Establish the shared type vocabulary used by both Core and the engine canister. Zero behavior, pure type definitions. All types must be Candid-compatible (no closures, no `Map.Map`, no `var`).

**Create:**

- `types/execution.mo` (importable by both Core and the engine actor class)
  - `ScopeResource` — variant: `{ #workspaces; #agents; #secrets; #eventStore; #sessionPolicy }`
  - `ScopeAccess` — variant: `{ #read; #write }` (write implies read)
  - `ScopeGrant` — `{ resource : ScopeResource; access : ScopeAccess }`
  - `ExecutionEnvelope` — `{ envelopeId : Text, envelopeVersion, requestId, agentId, agentName, workspaceId, workflowId, messages, instructions, constraints, secrets, scopeGrants, envelopeNonce : Text }`
  - `ExecutionConstraints` — `{ maxRounds : Nat; maxTokenBudget : ?Nat }`
  - `ExecutionStats` — `{ durationNs, llmCalls, toolCalls, inputTokens, outputTokens, model, rounds }`
  - `ExecutionStatus` — `{ #completed; #failed : Text; #roundLimitReached }`
  - `ExecutionResult` — `{ status : ExecutionStatus; stats : ExecutionStats }` — lightweight return from engine to orchestrator
  - `ChatMessage`, `ChatRole` — engine-local message types
  - `SummarizedStep` — `{ tool : Text; summary : Text; success : Bool }` — lightweight per-step detail for event payloads
  - `ExecutionEvent` — variant:
    - `#executionMilestone : { humanSummary : Text; stepsDetail : [SummarizedStep] }`
    - `#executionComplete : { humanSummary : Text; stepsDetail : [SummarizedStep]; status : ExecutionStatus; stats : ExecutionStats }`

**Design decisions:**

- **All types are Candid-serializable.** They cross the canister boundary (Core ↔ engine). No closures, no `Map.Map`, no mutable fields.
- **`envelopeId` identifies the execution request.** Each envelope gets a unique ID (generated by the orchestrator). Multiple executions can happen within the same turn (e.g., sub-agent delegation), so the `envelopeId` — not the `turnId` — is the correlation key for milestones and completion. The token nonce is uniquely bound to one envelope ID.
- **`envelopeId` identifies the execution request.** Each envelope gets a unique ID (generated by the orchestrator). Multiple executions can happen within the same turn (e.g., sub-agent delegation), so the `envelopeId` — not the `turnId` — is the correlation key for milestones and completion. The token nonce is uniquely bound to one envelope ID.
- **`envelopeNonce` in the envelope.** The engine needs the token nonce to authenticate every inter-canister call back to Core. Baked into the envelope at construction time. The engine already knows Core's principal from its constructor argument (`coreId`).
- **No `TraceEntry` or `TraceDetail`.** Traces are the engine's internal concern — they don't cross the canister boundary. The `SummarizedStep` in events is the only execution detail that reaches Core.
- `SummarizedStep` is intentionally lightweight — tool name + human-readable summary + success flag. Designed for both Slack updates and future trace inspection UI.
- `secrets` field: `{ apiKeys : [(Text, Text)] }` — flat key-value pairs.
- `scopeGrants` field: declarative — tells the engine which tools to offer and which Core API methods it's authorized to call.
- **No `stateSnapshots`** — state is accessed on-demand through inter-canister calls to Core.

**Checkpoint:** `icp build control-plane-core` — types compile, no consumers yet.

**Estimated scope:** ~80–120 lines.

## Phase 3 — Execution Token + Core's Execution API ✅ COMPLETE

**Goal:** The authorization layer (token service on Core) and the single `shared` endpoint + `ExecutionApiService` that the engine canister calls for effects and events.

### 3a. `services/execution-token-service.mo` — token lifecycle (Core-side) ✅

**Implemented:**

- `TokenStore` — `{ var nextTokenId : Nat; tokens : Map<Text, TokenRecord> }`
- `TokenRecord` — lean, session-context-free: `{ nonce : Text; envelopeId : Text; turnId : Text; grants : [ScopeGrant]; createdAtNs : Int; expiresAtNs : Int; var revoked : Bool }`
- `issue(store, envelopeId, turnId, grants) : Text` — generates nonce from monotonic counter, stores record (TTL from `Constants.EXECUTION_TOKEN_TTL_NS` = 60 min), calls `cleanup()` first
- `validate(store, nonce, requiredGrant : ScopeGrant) : Bool` — checks existence, revocation, expiry, and grant coverage
- `getRecord(store, nonce) : ?TokenRecord` — returns `null` for expired or revoked tokens
- `revoke(store, nonce)` — sets `revoked := true`
- `cleanup(store)` — removes expired/revoked tokens; called at the start of each `issue`

**`ScopeGrant` semantics (implemented in `types/execution.mo`):**

```motoko
public type ScopeGrant = {
  #workspace : { id : ?Nat; access : ScopeAccess };
  #agent : { id : ?Nat; access : ScopeAccess };
};

```

- `id = ?someId` — grants access to that specific resource
- `id = null` — grants create-only access (used for `POST /workspace` and `POST /agent`)
- `#write` access covers `#read` at the same id (any access level covers `#read`)
- Grant matching: resource variant must match AND `id` must match AND `coversAccess(held, required)` must pass

**Deferred from original plan:**

- No `sessionId`, `agentId`, `workspaceId`, `slackContext` on the token — context carried separately by Phase 6 orchestrator
- No `lastMilestonePostNs` — milestone debounce deferred to Phase 6
- No `derive()` / sub-agent delegation — deferred to future PR
- No `parentNonce` / `subAgentRequestId` — deferred to future PR

**`types/execution.mo` additions (also implemented here):**

- `HttpMethod = { #get; #post; #delete }` — Candid-native variant (no string parsing)
- `AsyncEffect = { #milestone{envelopeId;turnId;humanSummary;stepsDetail} | #complete{envelopeId;turnId;humanSummary;stepsDetail;status;stats} }` — replaces the planned event queue
- `HandleResult = { response : {#ok:Text;#err:Text}; asyncEffects : [AsyncEffect] }` — what `handleRequest` returns

### 3b. `services/execution-api-service.mo` — REST routing + handler dispatch ✅

**`handleRequest` signature:**

```motoko
public func handleRequest(
  method : ExecutionTypes.HttpMethod,
  path : Text,
  body : Text,
  deps : ServiceDeps,
) : ExecutionTypes.HandleResult;

```

`ServiceDeps = { tokenStore; workspaces; agentRegistry }` — no event queue.

**REST routing table (path parsed via `parsePath` → splits on `/`, filters empty segments):**

| Method   | Path               | Scope required                      | Handler                                                                 |
| -------- | ------------------ | ----------------------------------- | ----------------------------------------------------------------------- |
| `GET`    | `/workspace`       | `#workspace{id=?id;#read}` per item | List; filters to only workspaces the token can read                     |
| `GET`    | `/workspace/{id}`  | `#workspace{id=?id;#read}`          | Get single workspace                                                    |
| `POST`   | `/workspace`       | `#workspace{id=null;#write}`        | Create workspace                                                        |
| `POST`   | `/workspace/{id}`  | `#workspace{id=?id;#write}`         | Rename/update workspace                                                 |
| `DELETE` | `/workspace/{id}`  | `#workspace{id=?id;#write}`         | Delete workspace                                                        |
| `GET`    | `/agent`           | `#agent{id=?id;#read}` per item     | List; filters to only agents the token can read                         |
| `GET`    | `/agent/{id}`      | `#agent{id=?id;#read}`              | Get single agent                                                        |
| `POST`   | `/agent`           | `#agent{id=null;#write}`            | Create agent (`#custom`, engines=`[#canister]`)                         |
| `POST`   | `/agent/{id}`      | `#agent{id=?id;#write}`             | Update agent name/model                                                 |
| `POST`   | `/event/milestone` | token must exist + not expired      | Enqueue `AsyncEffect#milestone`; return `#ok` immediately               |
| `POST`   | `/event/complete`  | token must exist + not expired      | Revoke token + enqueue `AsyncEffect#complete`; return `#ok` immediately |

**Response shape:** `HandleResult.response = {#ok:Text;#err:Text}` — Candid variant, not JSON-wrapped.

**Async effects instead of an event queue:**

`handleRequest` returns `asyncEffects : [AsyncEffect]` in the `HandleResult`. `main.mo` schedules `Timer.setTimer<system>(#seconds 0, ...)` per effect after calling `handleRequest`. No dedicated event queue module exists — effects are processed inline in `processExecutionAsyncEffect` (stub until Phase 6).

**`event.complete` revokes the token synchronously** (before enqueuing), as defense-in-depth.

**Milestone debounce is NOT in the service** — `POST /event/milestone` always enqueues. Debounce logic deferred to Phase 6 when `processExecutionAsyncEffect` is wired to Slack.

**Transport-level guard in `main.mo`:** When `internalEnginePrincipal` is set, rejects calls from any other principal. When null (not yet set), allows all callers — enables testing before the engine canister is spawned.

## Phase 4 — Engine: Tool System

**Goal:** The engine's own tool registry, executor, and handler implementations. Tools call Core's execution API (`callCore` helper) for state operations via inter-canister calls.

**Create:**

- `src/internal-engine/tool-types.mo` — engine-local tool types
  - `ToolResult`, `ToolHandler` types
  - Handlers receive a `callCore` function (wrapper around `core.executionApi`) + token nonce
- `src/internal-engine/tool-registry.mo` — workflow-based tool selection
  - `getTools(workflowId, scopeGrants) → [Tool]` — returns tool definitions for a workflow, gated by which scopes are granted (checks `scopeGrants` array — if `#workspaces` not in grants, workspace tools are omitted)
  - `getHandler(workflowId, toolName) → ?ToolHandler` — returns the handler for a specific tool
  - Workflow `"admin-v1"`: same tools as current `FunctionToolRegistry.getAll(resources)`
  - Workflow `"onboarding-v1"`: stub, returns empty tool set
- `src/internal-engine/tool-executor.mo` — tool dispatch
  - `execute(callCore, workflowId, toolCalls) → async [ToolExecResult]` — executes tool calls through Core's API
  - `formatResultsForLlm(results) → Text`
- `src/internal-engine/handlers/` — tool handler modules
  - Port each handler from `tools/handlers/` into the engine's handlers directory
  - **Read handlers**: `list_workspaces` calls `await callCore(nonce, #get, "/workspace", obj([]))`
  - **Write handlers**: `create_workspace` calls `await callCore(nonce, #post, "/workspace", obj([("name", str(name))]))`
  - Each handler: parse JSON arguments → `await callCore(nonce, method, path, body)` → return result
  - Handlers are SIMPLER than current versions — no state refs, no closures, just a single API call

**Key differences from closure-based design:**

- **Tool gating uses `scopeGrants`** from the envelope (an `[ScopeGrant]` array) instead of null-checking API record fields. The registry checks `Array.find(grants, func(g) { switch g { case (#workspace(w)) { true }; case _ { false } } })` to decide whether to include workspace tools.
- **Every tool call is an `await`** (inter-canister call) — all handler functions are `async`. This is unavoidable with a separate canister.
- **If Core rejects a call** (wrong scope, expired token), the handler receives a trap or error and returns it to the LLM as a failed tool result. The engine continues (fault recovery).
- **Multi-round consistency**: round 2's reads see round 1's writes because Core operates on canonical state.
- **`try/catch` required**: inter-canister calls can fail for infrastructure reasons (Core upgrading, cycles exhausted). Handlers must catch and return graceful errors.

**Pragmatic approach for this PR:**

- **High-value tools first**: port the most-used admin tools (workspace CRUD, agent CRUD, secret management)
- **Stub remaining tools**: return "not yet available in this engine version" for rarely-used tools
- **Follow-up PR**: complete the tool migration

**Checkpoint:** `icp build` — tool system compiles within the engine source tree, tools can be invoked in isolation with a mock Core.

**Estimated scope:** ~600–1,000 lines (registry + executor + handler ports). Handlers are simpler than current versions since they delegate to Core's API.

## Phase 5 — Engine Actor Class + Loop Assembly

**Goal:** The engine canister actor class that ties together the LLM wrapper (Phase 2) and tool system (Phase 4) into the complete multi-round execution loop. This is a separate canister, spawned by Core.

**Create:**

- `src/internal-engine/main.mo` — the actor class

```motoko
import Principal "mo:core/Principal";
import Time "mo:core/Time";
import Types "../control-plane-core/types/execution";
import CoreApi "core-api";
import LlmWrapper "llm-wrapper";
import ToolExecutor "tool-executor";
import ToolRegistry "tool-registry";

persistent actor class InternalEngine(coreId : Principal) = self {

  let core : CoreApi.CoreApi = actor (Principal.toText(coreId));

  // Convenience wrapper — all engine-to-Core communication goes through this.
  // method + path map to REST routes; body contains envelopeNonce + payload fields.
  func callCore(envelopeNonce : Text, method : { #get; #post; #delete }, path : Text, body : JSON) : async {
    #ok : Text;
    #err : Text;
  } {
    let bodyWithToken = JSON.show(obj([("envelopeNonce", str(envelopeNonce))] # JSON.entries(body)));
    await core.executionApi(method, path, bodyWithToken);
  };

  // Example usages:
  //   await callCore(nonce, #get, "/workspace", obj([]))
  //   await callCore(nonce, #post, "/workspace", obj([("name", str("my-ws"))]))
  //   await callCore(nonce, #post, "/event/complete", obj([("humanSummary", str("Done")), ...]))

  // Transform function for HTTP outcalls — defined on the engine actor
  // Resolves the Phase 2 challenge: engine is its own actor, can define shared query
  public query func transform(raw : /* IC transform args */) : async /* IC response */ {
    // Strip non-deterministic headers, canonicalize body for consensus
  };

  // The single entry point for execution
  public shared ({ caller }) func execute(
    envelopeNonce : Text,
    envelope : Types.ExecutionEnvelope,
  ) : async Types.ExecutionResult {
    // Only Core can dispatch executions
    assert (caller == coreId);

    let apiKey = /* extract from envelope.secrets */;
    let tools = ToolRegistry.getTools(envelope.workflowId, envelope.scopeGrants);
    var messages = envelope.messages;
    var rounds : Nat = 0;
    var steps : [Types.SummarizedStep] = [];
    let startNs = Time.now();
    // ... stats accumulators (inputTokens, outputTokens, llmCalls, toolCalls)

    label loop_ loop {
      if (rounds >= envelope.constraints.maxRounds) {
        ignore await callCore(
          envelopeNonce,
          #post,
          "/event/complete",
          obj([
            ("humanSummary", str("Reached maximum rounds")),
            ("stepsDetail", arr(stepsToJson(steps))),
            ("status", str("roundLimitReached")),
            ("stats", statsToJson(buildStats(startNs, rounds))),
          ]),
        );
        return {
          status = #roundLimitReached;
          stats = buildStats(startNs, rounds);
        };
      };

      let result = await LlmWrapper.reason(apiKey, messages, model, instructions, temperature, tools);
      rounds += 1;

      switch (result) {
        case (#ok(#textResponse { text })) {
          let stats = buildStats(startNs, rounds);
          ignore await callCore(
            envelopeNonce,
            #post,
            "/event/complete",
            obj([
              ("humanSummary", str(text)),
              ("stepsDetail", arr(stepsToJson(steps))),
              ("status", str("completed")),
              ("stats", statsToJson(stats)),
            ]),
          );
          return { status = #completed; stats };
        };
        case (#ok(#toolCalls(calls))) {
          let results = await ToolExecutor.execute(callCore, envelopeNonce, envelope.workflowId, calls);
          // Accumulate SummarizedStep entries from results
          // Optionally emit milestone if meaningful work was done
          if (shouldEmitMilestone(results)) {
            try {
              ignore await callCore(
                envelopeNonce,
                #post,
                "/event/milestone",
                obj([
                  ("humanSummary", str(summarize(results))),
                  ("stepsDetail", arr(stepsToJson(steps))),
                ]),
              );
            } catch (e) {
              // Milestone emission failure is non-fatal — continue execution
            };
          };
          // Add tool results to messages, continue loop
        };
        case (#err(errMsg)) {
          let stats = buildStats(startNs, rounds);
          ignore await callCore(
            envelopeNonce,
            #post,
            "/event/complete",
            obj([
              ("humanSummary", str(errMsg)),
              ("stepsDetail", arr(stepsToJson(steps))),
              ("status", str("failed")),
              ("stats", statsToJson(stats)),
            ]),
          );
          return { status = #failed(errMsg); stats };
        };
      };
    };
  };
};

```

**Design decisions:**

- **The engine is a `persistent actor class`.** Core calls `await (with cycles) InternalEngine.InternalEngine(Principal.fromActor(Core))` to spawn it. The engine receives Core's principal and uses it to construct an inter-canister reference.
- **Caller validation**: `execute` asserts `caller == coreId`. Only Core can dispatch executions. This prevents unauthorized entities from triggering work on the engine.
- **Stateless between executions.** The engine has no meaningful persistent state — no sessions, no workspace data, no user data. It's a pure executor. The only persistent data is the `coreId` from the constructor.
- **`callCore` wrapper.** All engine-to-Core communication goes through a single helper that calls `core.executionApi(method, path, bodyWithToken)`. Tool handlers use REST-style paths (e.g., `#get "/workspace"`, `#post "/workspace/{id}"`). Event emission uses `#post "/event/complete"` and `#post "/event/milestone"`.
- **Transform function lives on the engine actor.** The engine defines its own `shared query func transform` — cleanly resolves the Phase 2 challenge.
- **Milestone selectivity**: the engine decides WHEN to emit `#executionMilestone`. Good candidates: write operations that changed state, tool rounds with 3+ tools. Core decides WHETHER to surface it (15s debounce).
- **Milestone emission is non-fatal.** `try/catch` around milestone calls — if Core is briefly unavailable, the engine continues. `event.complete` failure IS fatal (the engine returns `#failed`). Note: since `handleRequest` is sync and events are enqueued, the `callCore` for events returns `{ "ok": true }` quickly. The actual Slack post happens asynchronously in Core's queue processor.
- **Stats accumulation is local.** Token counts, duration, round count tracked in the engine, returned in `ExecutionResult`.
- **`try/catch` on Core calls.** Inter-canister calls can fail. Tool execution failures are caught and returned as failed tool results. `#executionComplete` failures are propagated.

**Checkpoint:** `icp build` (both Core and engine) + unit tests:

- Engine rejects calls from non-Core callers
- Mock LLM → `#textResponse`: `callCore("event.complete", ...)` called, result has `#completed`
- Mock LLM → `#toolCalls` then `#textResponse`: tool calls go through `callCore`, milestone may be emitted
- Mock LLM → loop `maxRounds`: result has `#roundLimitReached`, `event.complete` emitted
- Mock LLM → `#err`: result has `#failed`, `event.complete` emitted with error
- Core call failure during tool execution: engine returns `#failed` gracefully

**Estimated scope:** ~200–350 lines (actor class + loop logic).

## Phase 6 — Engine Lifecycle + Orchestrator Rewiring

**Goal:** Core manages the engine canister lifecycle (spawn, top-up, upgrade propagation) and the orchestrator dispatches to the engine via inter-canister calls.

### 6a. Engine lifecycle in `main.mo`

**Persistent state:**

```motoko
var engine : ?InternalEngine.InternalEngine = null;
var lastEngineTopUpTimestamp : Int = Time.now();

```

**Spawn (lazy init):**

```motoko
func deployEngine() : async InternalEngine.InternalEngine {
  await (with cycles = 1_000_000_000_000) InternalEngine.InternalEngine(Principal.fromActor(Core));
};

func ensureEngine() : async InternalEngine.InternalEngine {
  switch (engine) {
    case (?e) { e };
    case null {
      let e = await deployEngine();
      engine := ?e;
      e;
    };
  };
};

```

**Top-up runner** — new timer runner following the existing runner pattern:

- `timers/engine-topup-runner.mo`
  - `run(engineOpt : ?InternalEngine.InternalEngine) : async { #ok; #err : Text }`
  - Calls `IC.canister_status` on the engine canister
  - If `status.cycles < 500_000_000_000`, deposits 1T cycles via `IC.deposit_cycles`
  - Returns `#ok` on success, `#err` if engine is null or call fails

Timer registry entry (added to `timerRegistry()` in `main.mo`):

```motoko
{
  name = "engine-topup";
  interval = Constants.SEVEN_DAYS_NS;
  getLastRun = func() : Int { lastEngineTopUpTimestamp };
  setLastRun = func(t : Int) { lastEngineTopUpTimestamp := t };
  wrappedRun = func() : async { #ok; #err : Text } {
    EngineTopUpRunner.run(engine);
  };
};

```

**Upgrade propagation** — on Core's `postupgrade`:

```motoko
system func postupgrade() {
  // ... existing timer rescheduling ...

  // Schedule engine upgrade 1 minute after Core upgrade
  ignore Timer.setTimer<system>(
    #seconds 60,
    func() : async () {
      switch (engine) {
        case null {};
        case (?e) {
          await (system InternalEngine.InternalEngine)(#upgrade e)(
            Principal.fromActor(Core)
          );
        };
      };
    },
  );
};

```

This ensures the engine canister always runs the same Wasm version as Core (the actor class is compiled into Core's Wasm). The 1-minute delay gives Core time to stabilize after upgrade.

### 6b. `orchestrators/agent-orchestrator.mo`

Current flow:

```
orchestrateAgentTalk(agent, ...) → resolve API key → OrgAdminAgent.process(agent, ...) → return response
```

New flow:

````
orchestrateAgentTalk(agent, ...) →
  1. Resolve API key (guard rail — stays here)
  2. Resolve Slack context: botToken (decrypt), channel, threadTs from incoming msg
  3. Determine scope grants for the agent (from agent config / workflow)
  4. Generate envelopeId (e.g., "env_" # Nat.toText(nextEnvelopeId))
  5. Issue execution token: `ExecutionTokenService.issue(tokenStore, envelopeId, turnId, grants)` — returns `envelopeNonce`; Slack context and session/agent/workspace bindings are stored separately in the orchestrator and passed to `processExecutionAsyncEffect` via closure when the timer fires
  6. Ensure engine canister exists: ensureEngine()
  7. Build ExecutionEnvelope:
     - envelopeId
     - InstructionComposer.compose() → instructions
     - ContextAssembler.assemble() → messages
     - constraints from agent config or defaults
     - secrets: { apiKeys: [("openrouter", apiKey)] }
     - workflowId (default: "admin-v1")
     - scopeGrants, envelopeNonce
  8. Set turn status to #pending (turn awaits milestones + final delivery)
  9. Call engine with try/catch (see below)
  10. Revoke token (belt-and-suspenders — event.complete already revokes)
  11. Return result to caller

Step 6 — engine call with error handling:

```motoko
try {
  let result = await engine.execute(envelopeNonce, envelope);
  // success — proceed to step 7
} catch (e) {
  switch (Error.code(e)) {
    case (#system_transient) {
      // Engine may be frozen (low cycles) or subnet transient error.
      // Check canister_status to confirm and top up if needed.
      try {
        let status = await IC.canister_status({ canister_id = Principal.fromActor(engine) });
        if (status.cycles < 500_000_000_000) {
          let topUpAmount = 1_000_000_000_000; // 1T
          Cycles.add<system>(topUpAmount);
          await IC.deposit_cycles({ canister_id = Principal.fromActor(engine) });
        };
      } catch (_) {};
      // Return a retriable failure to the orchestrator
    };
    case (#canister_error) {
      // Engine trapped — likely a bug in the engine code
    };
    case _ {
      // Other errors (canister_reject, etc.)
    };
  };
  // All error paths: revoke token, return #failed with Error.message(e)
};
````

```

**No `ExecutionApiBuilder`.** Core exposes a single `executionApi` endpoint backed by `ExecutionApiService`. The token + scope grants handle authorization inside the service.

**Turn lifecycle:** The turn transitions to `#pending` before dispatching to the engine. The turn stays in `#pending` until the engine sends `POST /event/complete`, which triggers an `AsyncEffect#complete` that is scheduled by `main.mo` via `Timer.setTimer` and processed in `processExecutionAsyncEffect` (calls `SessionModel.completeTurn()` → `#succeeded` or `#failed`). If the engine call itself fails (catch block), the orchestrator completes the turn as `#failed` directly.

**`TurnStatus` change:** Add `#pending` variant to `TurnStatus` in `session-model.mo`: `{ #running; #pending; #succeeded; #failed }`. `#pending` means "dispatched to engine, awaiting milestones and completion."

**Every inter-canister call to the engine is wrapped in `try`/`catch`.** On `#system_transient`, Core checks the engine's `canister_status` and tops up if cycles are low — this catches cycle exhaustion mid-flight rather than waiting for the next top-up runner tick. On `#canister_error`, the engine trapped (bug). All error paths revoke the token and return `#failed` with the error message.
```

```
## Risk Register

| Risk                                                      | Impact                                                                         | Mitigation                                                                                                              |
| --------------------------------------------------------- | ------------------------------------------------------------------------------ | ----------------------------------------------------------------------------------------------------------------------- |
| Inter-canister call latency per tool operation            | Medium — each tool read/write is a round-trip (~1–2 rounds on same subnet)     | Tools are already async. Batch reads where possible. Same-subnet calls are fast (~200ms).                               |
| Engine canister runs out of cycles                        | High — execution fails mid-way                                                 | Every call to engine wrapped in `try`/`catch`. On `#system_transient`: check `canister_status`, top up if cycles < 500B. Weekly top-up runner as baseline. `ensureEngine()` pre-checks on spawn. |
| Engine LLM wrapper diverges from original → bugs          | High — HTTP outcalls are consensus-sensitive                                   | Copy-adapt approach; diff against original to verify no logic drift                                                     |
| Partial state mutation on engine failure                  | Medium — if engine fails mid-execution, some Core state writes already applied | Same trade-off as current system. Add compensation/rollback if needed (future).                                         |
| Tool handler migration scope is larger than estimated     | Medium — delays Phase 4                                                        | Port high-value tools first; stub low-frequency tools.                                                                  |
| Token store grows unbounded                               | Low — cleanup on each `issue()` removes expired/revoked tokens                 | TTL + explicit revocation + periodic cleanup                                                                            |
| Agent router return type changes break downstream         | Medium — affects agent-router, message-handler, integration tests              | Minimize blast radius: orchestrator can unwrap ExecutionResult into existing return shape                               |
| Cassette re-recording needed for all integration tests    | Low — mechanical but time-consuming                                            | Re-record only affected tests                                                                                           |
| 2MB payload limit on inter-canister calls                 | Low — envelopes are mostly text (messages + instructions)                      | Monitor payload sizes. Chunk large context if needed. Most envelopes are well under 1MB.                                |
| Unbounded wait on `engine.execute()` blocks Core upgrades | Medium — if engine hangs, Core can't stop until the call completes             | Engine has `maxRounds` constraint. Token TTL (1h) is the ultimate safety net. Future: bounded wait calls.               |
| Upgrade timing mismatch (Core upgrades, engine still old) | Low — 1-minute delay timer propagates upgrade                                  | Timer fires quickly. If engine call is in flight during upgrade, it completes against old Wasm, next call uses new.     |
| Engine spawning fails (insufficient cycles)               | Medium — first execution after init fails                                      | `deployEngine()` attaches 1T cycles. Top-up runner maintains balance. `ensureEngine()` is called before every dispatch. |
```
